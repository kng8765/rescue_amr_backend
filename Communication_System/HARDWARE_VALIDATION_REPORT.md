# Hardware Validation Report

## 1. Purpose

본 문서는 구조 로봇 음성 통신 시스템에 사용할 Android 모바일 또는 태블릿의 초기 하드웨어 검증 결과를 기록한다.

검증 목적은 관제 PC와 Android 장치 간 실시간 음성 통신 구현 전에, 실제 장치 조건이 프로젝트 요구사항을 만족하는지 확인하는 것이다.

## 3. Detailed Results

### 3.1 Android Device Model

현재 사용할 Android 태블릿 또는 모바일의 구체적인 기종은 확정되지 않았다.

영향:

- WebRTC SDK 호환성 검증이 아직 불가능하다.
- 마이크, 스피커, 배터리, 발열, 화면 유지 정책 검증이 제한된다.
- APK 설치 및 권한 정책 확인이 장치별로 달라질 수 있다.

필요 작업:

- 실제 사용할 장치 모델명을 확정한다.
- 동일 모델로 반복 테스트를 수행한다.
- 가능하면 최종 시연 장비와 개발 테스트 장비를 동일 모델로 맞춘다.

### 3.2 Android Version

확인된 Android 버전은 Android 16이다.

판정:

- 기존 요구사항인 Android 11 이상 조건을 만족한다.
- WebRTC 기반 음성 통신 앱 개발 대상 버전으로 사용할 수 있다.

주의사항:

- Android 16의 마이크 권한, 백그라운드 실행 제한, 배터리 최적화 정책을 별도로 확인해야 한다.
- 화면 항상 켜짐, 앱 자동 실행, WiFi 자동 재연결 설정 가능 여부를 실제 장치에서 검증해야 한다.

### 3.3 WiFi Performance

확인된 WiFi 성능은 245 Mb/s, 5.3 GHz이다.

판정:

- 음성 통신에는 충분한 대역폭이다.
- WebRTC Opus 음성 통신은 일반적으로 매우 낮은 대역폭으로도 동작하므로, 대역폭 자체는 초기 구현의 병목이 아닐 가능성이 높다.

주의사항:

- 245 Mb/s는 링크 속도 또는 측정 환경 기준일 수 있으므로 실제 현장 처리량과 지연 시간은 별도 측정해야 한다.
- 구조물, 거리, 로봇 이동, AP 위치에 따라 WiFi 품질이 크게 달라질 수 있다.
- Phase 2에서 Ping Test, Bandwidth Test, WiFi 약화 테스트가 필요하다.

### 3.4 Microphone and Speaker

현재 계획은 모바일 내장 마이크와 내장 스피커를 사용하는 것이다.

판정:

- 초기 Prototype에는 사용 가능하다.
- 현장 운용 가능 여부는 실측이 필요하다.

위험 요소:

- 로봇 모터 소음이 마이크에 크게 유입될 수 있다.
- 구조 대상자와 장치 사이 거리가 멀면 음성 수음 품질이 낮아질 수 있다.
- 내장 스피커 출력이 현장 소음보다 작을 수 있다.
- Full Duplex 통화 중 스피커 출력이 마이크로 재입력되어 Echo 또는 하울링이 발생할 수 있다.

필요 작업:

- 3m, 5m, 10m 거리 음성 테스트를 수행한다.
- 로봇 모터 동작 중 음성 테스트를 수행한다.
- WebRTC Echo Cancellation, Noise Suppression, Automatic Gain Control 적용 결과를 비교한다.
- 필요 시 외장 마이크 또는 외장 스피커 사용을 검토한다.

### 3.5 Robot Device Placement

로봇 본체에 별도 스피커와 마이크를 부착하기 어려우므로, Android 모바일 또는 태블릿을 후발 로봇 위에 올려놓고 운용하는 방식으로 결정했다.

판정:

- 로봇 본체 개조 없이 초기 음성 통신 Prototype을 진행할 수 있다.
- Android 내장 마이크와 내장 스피커를 그대로 사용한다.
- 후발 로봇과 Android 단말의 물리적 위치는 대체로 일치하므로 구조 대상자가 로봇 근처에서 관제 PC와 대화하는 시나리오를 유지할 수 있다.

영향:

- 주행 중 단말이 흔들리거나 떨어질 수 있다.
- 로봇 진동과 모터 소음이 마이크에 유입될 수 있다.
- 단말 방향에 따라 스피커 출력과 마이크 수음 품질이 달라질 수 있다.
- 장시간 운용 시 배터리와 화면 유지 설정을 별도로 관리해야 한다.

필요 작업:

- 미끄럼 방지 패드, 임시 고정 밴드, 보호 케이스 중 사용할 방식을 정한다.
- 로봇 정지 상태와 이동 상태에서 3m, 5m, 10m 음성 테스트를 수행한다.
- 단말 방향별 마이크 수음 품질과 스피커 출력 품질을 비교한다.
- 충전 케이블 연결 여부와 배터리 운용 시간을 확인한다.

권장 대응:

- 1차 권장안: 후발 로봇 상단에 미끄럼 방지 패드와 보호 케이스를 사용한다.
- 2차 대안: 주행 테스트 중 낙하 위험이 있으면 임시 고정 밴드를 추가한다.
- 3차 대안: 내장 마이크와 스피커 품질이 부족하면 외장 음향 장치를 재검토한다.

## 5. Conclusion

현재 Android 버전과 WiFi 성능은 초기 요구사항을 만족한다. 따라서 PC와 Android 장치 간 WebRTC 음성 통신 Prototype을 개발하기 위한 기본 소프트웨어 조건은 확보된 상태이다.

다만 Android 장치의 기종은 아직 미정이다. 또한 단말을 후발 로봇 위에 올려놓는 운용 방식은 확정되었지만, 주행 중 안정성, 전원 공급, 단말 방향, 마이크 및 스피커 품질은 실측이 필요하다.

다음 단계에서는 동일 WiFi 네트워크에서 관제 PC Signaling Server를 실행하고, PC와 Android 단말 간 WebRTC 음성 통신 Prototype을 구축한다.
